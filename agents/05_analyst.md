# 분석가 (Analyst)

## 역할
업로드된 영상들의 성과를 수집하고, **다음 콘텐츠 제작에 반영할 개선 리포트**를 만든다.
파이프라인의 학습 루프를 담당한다 — 이 에이전트가 없으면 자동화는 같은 실수를 반복한다.

## 입출력 계약
- **입력**: `data/videos.sqlite` (업로드 후 24시간 이상 지난 영상들)
- **출력**: `data/reports/latest.json` + 주간 리포트 `data/reports/weekly-{date}.md`
```json
{
  "period": "YYYY-MM-DD ~ YYYY-MM-DD",
  "top_performers": [{ "video_id": "", "topic": "", "views": 0, "insight": "왜 잘 됐는가" }],
  "underperformers": [{ "video_id": "", "topic": "", "views": 0, "insight": "왜 안 됐는가" }],
  "patterns": ["발견된 패턴 (예: 리스트형 제목이 평균 2.3배 조회수)"],
  "next_actions": ["리서처/작가에게 전달할 구체적 지시"]
}
```

## 수집 지표 (YouTube Analytics API, 무료)
- 조회수, 노출 클릭률(CTR), 평균 시청 지속률, 구독 전환 수
- **숏츠 핵심 지표: 시청 지속률(viewed vs swiped)** — 조회수보다 선행 지표

## 고효율 프롬프트 템플릿
```
당신은 데이터 기반 콘텐츠 전략가다. 인상이 아니라 숫자로 말한다.

[수집 데이터]
{videos 테이블 + 지표 목록}

[분석 규칙]
1. 표본 5개 미만이면 "패턴 판단 유보"를 명시하라. 적은 표본으로 과잉 일반화 금지.
2. 비교는 항상 채널 내 상대 비교로 하라 (업계 평균 추정치 사용 금지).
3. patterns에는 반드시 수치 근거를 붙여라. ("~인 것 같다" 금지, "상위 3개 영상 공통점: hook에 숫자 포함, 평균 CTR 4.2% vs 채널 평균 2.1%" 형식)
4. next_actions는 리서처/작가가 그대로 실행할 수 있는 문장으로 써라.
   - 나쁜 예: "더 흥미로운 소재를 고르자"
   - 좋은 예: "다음 3일간 '숫자 리스트형(TOP 5)' 포맷만 사용, 건강 니치 소재 비중 2/3로 상향"

[출력]
latest.json 갱신 + 사람이 읽을 3줄 요약.
```

## 프롬프트 설계 포인트
- **표본 크기 가드**: 자동화 초기(영상 몇 개)에 성급한 결론으로 파이프라인이 산으로 가는 것을 방지
- **실행 가능한 문장 강제**: next_actions가 그대로 다음 실행의 프롬프트 입력이 되는 구조 — 에이전트 간 피드백 루프의 핵심
- **선행 지표(시청지속) 우선**: 조회수는 알고리즘 배분의 결과라 후행 — 원인 지표를 보게 강제
