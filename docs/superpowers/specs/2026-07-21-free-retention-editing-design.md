# 무료 잔존율 우선 편집 개선 설계

## 목표

추가 유료 API 없이 기존 무료 미디어, Google TTS, FFmpeg 합성 구조를 유지하면서 Shorts 초반 이탈을 줄이고 본문 시청 흐름을 강화한다. 로컬 샘플 한 편으로 결과를 확인한 뒤에만 운영 서버에 반영한다.

## 범위

- 가장 강한 무료 소스를 첫 프레임부터 움직이게 표시하면서 짧은 주제 음성을 재생한다.
- 장면 역할에 따라 1.8~3.5초 단위로 쇼트를 분할한다.
- 정지 이미지와 비디오에 서로 다른 미세 확대·이동 패턴을 교차 적용한다.
- 자막을 기존 하단에서 영상 화면의 하단 중앙 영역으로 올린다.
- 핵심 단어와 숫자를 제한적으로 강조하고 숫자·단위의 중간 줄바꿈을 금지한다.
- hook 또는 payoff 전환에만 짧고 작은 자체 생성 효과음을 넣는다.
- 무료 미디어 선택 시 세로 비율, 해상도, 영상 여부, 중복 여부를 점수화하고 사용할 수 없는 파일은 다음 후보로 폴백한다.
- 모든 선택과 폴백 결과를 `produce_log.json`에 기록한다.

## 비범위

- Gemini Omni, Veo 등 유료 영상 생성 API
- 외부 유료 음원 또는 효과음
- 업로드 일정, 일일 업로드 수, 주제 카테고리 변경
- 모니터링 프론트 변경
- 샘플 승인 전 운영 서버 변경

## 편집 규칙

### 인트로와 쇼트 길이

- 첫 화면은 검은 화면이나 별도 타이틀 카드 없이 첫 번째 hook 미디어로 즉시 시작한다.
- 상단에는 전체 제목을 계속 고정한다.
- 인트로 음성은 전체 제목에서 핵심 주제만 추린 짧은 문구를 사용하며 목표 길이는 1.2~1.8초다.
- 첫 hook 쇼트는 1.8~2.2초, context/problem은 2.4~3.2초, mechanism은 2.2~3.0초, payoff는 2.0~2.8초, close는 최대 3.5초로 계획한다.
- 한 장면의 실제 음성이 계획보다 길면 동일 검색어의 다른 소스를 순환해 전체 음성을 덮는다.

### 화면 움직임

- 이미지는 좌우 패닝, 상하 패닝, 느린 확대 중 하나를 쇼트 번호에 따라 순환한다.
- 비디오는 중앙 고정만 사용하지 않고 1.00~1.04 범위의 미세 확대 또는 크롭 이동을 적용한다.
- 화면 전환 효과는 화려한 템플릿 대신 짧은 하드 컷을 기본으로 하며 payoff 직전만 최대 120ms의 짧은 디졸브를 허용한다.
- 상하단 검은 프레임과 상단 고정 제목은 그대로 유지한다.

### 자막

- 자막 중심은 전체 1920px 캔버스 기준 약 70% 높이, 실제 영상 영역 안의 하단 중앙에 둔다.
- ASS 정렬 기준과 여백을 고정해 한 줄과 두 줄 자막이 위아래로 튀지 않게 한다.
- 기본 크기는 현재보다 키우지 않는다.
- 한 큐에는 최대 두 줄만 허용한다.
- 숫자와 바로 뒤 단위, 공백 없는 한국어 토큰은 중간에서 나누지 않는다.
- 숫자, 고유명사, 반전어 중 큐당 최대 한 구간만 노란색 계열로 강조한다.
- 인트로와 CTA에도 같은 위치 규칙을 적용하되 CTA는 중복 생성하지 않는다.

### 오디오

- 현재 Google TTS 음성, 속도, 피치 설정을 유지한다.
- BGM은 현재 side-chain ducking을 유지하고 음성 구간에서는 음성보다 명확히 작게 들리게 한다.
- 효과음은 FFmpeg로 생성한 80~140ms 저음량 전환음만 사용한다.
- 모든 컷에 효과음을 넣지 않고 hook 시작 또는 payoff 진입 중 최대 두 번만 넣는다.
- 효과음 추가 후에도 최종 오디오 길이는 영상 길이와 일치해야 한다.

## 무료 미디어 선택

- 후보 순위는 세로 영상, 720p 이상, 영상 소스, 높은 해상도 순으로 평가한다.
- 같은 영상 ID는 한 편에서 다시 사용하지 않는다.
- 다운로드 실패, 길이 0초, 읽을 수 없는 파일은 후보를 폐기하고 같은 검색어의 다음 후보를 시도한다.
- 정확한 장소·구조물은 현재와 같이 허용 라이선스가 기록된 Wikimedia 이미지를 우선한다.
- 모든 후보가 실패하면 기존 fallback 이미지를 사용해 제작 자체는 중단하지 않는다.

## 데이터 흐름

1. writer가 기존 story 스키마로 장면과 역할을 생성한다.
2. producer가 역할별 쇼트 길이로 shot plan을 만든다.
3. media library가 점수와 중복 상태를 이용해 무료 후보를 선택한다.
4. producer가 쇼트별 움직임을 적용하고 인트로·본문·CTA를 연결한다.
5. 자막 강조와 새 위치를 적용하고 BGM·전환음을 혼합한다.
6. 기존 media probe로 길이, 해상도, 검은 화면, 오디오를 검증한다.
7. 결과와 편집 통계를 `produce_log.json`에 기록한다.

## 실패 처리

- 후보 다운로드나 검증 실패는 다음 후보로 이어지고 전체 run을 즉시 실패시키지 않는다.
- 전환음 생성 실패 시 전환음만 생략한다.
- 자막 강조 파싱 실패 시 일반 흰색 자막으로 렌더링한다.
- 최종 영상 검증 실패 시 업로드 단계로 넘기지 않는다.
- 운영 서버 반영 전에는 기존 예약 실행 파일과 서비스 설정을 변경하지 않는다.

## 테스트와 승인 기준

- 역할별 shot duration 단위 테스트
- 숫자·단위·공백 없는 토큰 줄바꿈 회귀 테스트
- 자막 위치와 강조 스타일 문자열 테스트
- 이미지·영상 모션 필터 단위 테스트
- 무료 후보 점수, 중복 제외, 손상 파일 폴백 테스트
- 효과음 최대 횟수와 오디오 믹스 명령 테스트
- 전체 테스트 통과 후 로컬 sample 생성
- sample은 1080x1920, 60~75초, 영상·오디오 길이 차 0.5초 이하, 검은 화면 비율 기준을 통과해야 한다.
- 사용자가 sample을 확인하기 전에는 운영 서버에 배포하지 않는다.

## 정리 정책

- `.analysis_reference/`는 목표 영상 비교 자료이므로 샘플 승인 전까지 보존하고 Git 추적에서는 제외한다.
- 기존 작업 worktree의 샘플·교체 영상은 새 sample 승인 후 별도 확인을 거쳐 정리한다.
- 재생성 가능한 다운로드 캐시와 Python 캐시만 정리 대상으로 취급한다.
