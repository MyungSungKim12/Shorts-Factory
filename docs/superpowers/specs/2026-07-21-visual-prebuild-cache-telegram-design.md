# 화면 일치도·사전 제작·검증 캐시·텔레그램 알림 설계

## 목표

무료 운영 범위에서 다음 네 문제를 함께 해결한다.

1. 고유 장소·구조물 소재에 관계없는 스톡 영상이 섞이는 문제
2. 업로드 시각에 리서치부터 시작해 정시 실패가 늦게 발견되는 문제
3. 무료 그라운딩 할당량 소진 시 검증 캐시가 비어 있는 문제
4. 제작·검증·업로드 실패를 사용자가 늦게 발견하는 문제

오늘 `20260721-2` 17시 회차는 이미 품질검사를 통과한 기존 완성본을 유지한다. 새 규칙은 배포 후 처음 시작되는 사전 제작 회차부터 적용한다.

## 1. 화면과 대본의 일치도

### 소재 시각 식별자

스토리 소재에는 다음 시각 식별 정보를 둔다.

- `exact_queries`: Wikimedia Commons에서 실제 대상을 찾을 검색어
- `safe_fallbacks`: 실제 대상을 구하지 못했을 때 허용할 동일 계열 일반 영상 검색어
- `required_exact`: 고유 장소·구조물처럼 실제 대상 자료가 최소 한 건 필요한지 여부

리서처가 값을 제공하지 못한 경우 `topic`, `target_keyword`, `visual_plan`에서 결정론적으로 구성한다. 최신 사실이나 새로운 수치를 만들지 않는다.

### 장면별 사용 규칙

- 인트로·훅·결론 중 최소 한 장면에는 `exact_queries`로 확보한 허용 라이선스 Wikimedia 이미지를 사용한다.
- 설명 장면은 해당 소재의 `safe_fallbacks` 또는 작가의 구체적 검색어를 사용할 수 있다.
- 고유 사막 소재에 달 표면·무관한 유적처럼 소재 계열 밖의 범용 대체어를 사용하지 않는다.
- `exact:` 검색은 Wikimedia를 우선하며 파일 제목과 핵심 검색 토큰이 겹치는 후보만 허용한다.
- `required_exact=true`인데 실제 대상 자료를 한 건도 확보하지 못하면 제작을 실패시킨다. 관계없는 영상으로 성공 처리하지 않는다.

`produce_log.json`에는 `visual_relevance`를 기록한다.

- `required_exact`
- `exact_source_count`
- `generic_fallback_count`
- `unrelated_fallback_count`(항상 0이어야 함)
- 실제 사용 검색어와 출처

## 2. 예약 회차 자동 사전 제작

### 일정

KST 기준으로 업로드 두 시간 전에 제작한다.

| 업로드 회차 | 사전 제작 | 업로드 |
|---|---:|---:|
| 1 | 09:00 | 11:00 |
| 2 | 15:00 | 17:00 |
| 3 | 19:00 | 21:00 |

사전 제작은 대상 회차를 명시적으로 받는다. 제작 종료 시각을 기준으로 다른 회차에 임의 승격하지 않는다.

### 실행 흐름

1. 전역 파이프라인 잠금을 획득한다.
2. `data/staging/{staging_id}`에서 리서치·대본·합성을 수행한다.
3. 영상·오디오·대본 해시·CTA·화면 관련성 품질검사를 수행한다.
4. 모두 통과하면 `data/work/{run_id}`로 원자적으로 승격한다.
5. 업로드 시각의 cron은 준비된 파일을 재사용하고 YouTube 업로드만 수행한다.

사전 제작이 실패하면 staging과 오류 로그를 보존한다. 업로드 시각에 준비된 완성본이 없을 때만 기존 정시 제작 파이프라인을 최후의 대안으로 실행한다. 준비 중인 프로세스와 정시 프로세스가 겹치면 전역 잠금으로 직렬화하되, 죽은 프로세스의 잠금만 회수한다.

## 3. 무료 검증 캐시 자동 축적

### 적재 정책

- 매일 06:30 KST에 캐시 워밍 작업을 실행한다.
- 회차 1·2·3 카테고리별 최대 한 건씩 새 소재를 시도한다.
- `grounded_search`에 성공하고 계약 검증을 통과한 소재만 저장한다.
- `model_memory` 또는 출처가 불완전한 소재는 검증 캐시에 저장하지 않는다.
- 무료 검색 할당량 소진이 확인되면 그날 워밍을 즉시 종료하고 재시도 폭주를 막는다.
- 최근 업로드·기존 캐시와 제목이 같은 소재는 저장하지 않는다.
- 목표는 회차별 10건, 총 30건이다. 목표에 도달한 회차는 추가 검색을 생략한다.
- 검증 후 30일이 지난 항목은 일반 재사용 대상에서 제외하고 향후 재검증 대상으로 남긴다.

기존 실행 파이프라인의 우선순위는 `grounded_search → verified_cache → 불변 소재에 한한 model_memory`를 유지한다.

## 4. 텔레그램 운영 알림

### 설정

서버 `.env`에만 다음 값을 둔다.

- `TELEGRAM_ALERTS_ENABLED=true`
- `TELEGRAM_BOT_TOKEN` — BotFather 발급 토큰
- `TELEGRAM_CHAT_ID` — 사용자가 확인한 개인 Chat ID

토큰은 Git, 로그, API 응답, 테스트 fixture에 기록하지 않는다. 토큰이나 Chat ID가 없으면 알림만 비활성화하고 영상 파이프라인은 계속 실행한다.

### 알림 이벤트

- 사전 제작 성공: 회차, 제목, 길이, 검증 방식, 품질검사 결과
- 사전 제작 실패: 회차, 실패 단계, 안전하게 축약한 오류
- 업로드 성공: 회차, 제목, YouTube Shorts URL
- 업로드 최종 실패: 재시도 횟수, 실패 단계, 오류
- 캐시 워밍 결과: 새 적재 수, 회차별 보유 수
- 무료 검색 일일 할당량 소진 및 회차별 캐시 부족
- 전역 잠금 대기 시간 초과 또는 프로세스 타임아웃

동일 이벤트 키는 일정 시간 중복 전송하지 않는다. Telegram HTTP 요청은 짧은 연결·응답 타임아웃을 사용하며 실패를 로그에만 남긴다. Telegram 장애는 제작·업로드 성공 여부를 바꾸지 않는다.

## 오류 처리와 데이터 안전

- 완성된 예약 회차가 존재하면 새 사전 제작이 덮어쓰지 않는다.
- 품질검사 실패 파일은 `data/work`로 승격하지 않는다.
- 이미 업로드된 회차는 사전 제작·승격·재업로드하지 않는다.
- 캐시 워밍은 YouTube 업로드를 호출하지 않는다.
- 알림 메시지는 토큰, 인증 경로, 전체 예외 응답을 포함하지 않는다.
- 기존 `.env`, `credentials/`, SQLite, work/staging/recovery는 배포 전에 백업한다.

## 검증 계획

- 정확 검색 토큰 일치·불일치 후보 선별 단위 테스트
- 고유 대상 자료 미확보 시 제작 차단 테스트
- 시각 관련성 로그 및 품질 게이트 테스트
- 09/15/19시 대상 회차 계산과 원자적 승격 테스트
- 준비본 존재·부재에 따른 정시 실행 분기 테스트
- 캐시 목표치·중복·30일 만료·할당량 소진 중단 테스트
- Telegram 성공·비활성·타임아웃·중복 억제·비밀값 비노출 테스트
- 전체 회귀 테스트와 서버 원격 테스트
- 다음 사전 제작 회차를 실제로 생성해 품질검사 후, 정시 업로드 URL과 DB 단일 행을 확인

## 배포 및 원복

소스·cron·운영 데이터 백업 후 코드와 cron을 함께 배포한다. 배포 직후 대시보드 헬스, cron 7개 항목(업로드 3, 사전 제작 3, 캐시 워밍 1), 전역 잠금 부재를 확인한다. 문제가 생기면 백업 코드와 기존 11/17/21 업로드 cron으로 되돌리며 이미 검증된 예약 영상과 업로드 이력은 보존한다.
