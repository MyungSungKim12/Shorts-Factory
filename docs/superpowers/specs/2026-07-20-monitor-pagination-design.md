## API 계약

### `GET /api/videos`

쿼리:

- `page`: 기본값 1, 최솟값 1
- `page_size`: 기본값 10, 최솟값 1, 최댓값 50

응답:

```json
{
  "videos": [],
  "pagination": {
    "page": 1,
    "page_size": 10,
    "total_items": 27,
    "total_pages": 3,
    "has_previous": false,
    "has_next": true
  }
}
```

SQLite에서 `status = 'uploaded'`인 현재 유효 영상만 대상으로 `COUNT(*)`를 구하고 `ORDER BY uploaded_at DESC, video_id DESC LIMIT ? OFFSET ?`로 현재 페이지만 조회한다. `replaced`·삭제 처리된 기록은 활성 영상 목록에서 제외한다. DB가 없으면 빈 배열과 0건 페이징 정보를 반환한다.

### `GET /api/history`

쿼리와 응답의 `pagination` 계약은 영상 목록과 같다. `runs` 배열만 현재 페이지에 해당한다. 로그 파일은 파일명 문자열이 아니라 JSON의 `timestamp`와 회차 번호를 기준으로 최신순 정렬한다. 손상된 JSON 파일은 제외하되 정상 파일의 페이징은 계속 제공한다.

페이지 범위를 넘어선 요청은 빈 목록과 요청된 페이지 정보를 반환한다. 프론트가 `total_pages`를 보고 마지막 유효 페이지로 이동한다.

## 프론트엔드 동작

- 영상과 실행 이력에 각각 독립된 `videoPage`, `historyPage` 상태를 둔다.
- 페이지당 10개를 요청한다.
- 각 목록 아래에 `처음`, `이전`, 근처 페이지 번호, `다음`, `마지막` 버튼을 표시한다.
- 전체 건수와 `현재 페이지 / 전체 페이지`를 함께 표시한다.
- 30초 자동 갱신과 수동 새로고침 모두 현재 페이지를 유지한다.
- 갱신 후 현재 페이지가 전체 페이지보다 크면 마지막 유효 페이지로 한 번 이동한다.
- 목록별 로딩, 오류, 빈 결과를 서로 다른 문구로 표시한다.
- API 응답이 성공이 아니면 오류 상태로 처리한다.
- 헤더의 실행 일정은 `매일 11:00·17:00·21:00 자동 실행`으로 표시한다.
- 표는 작은 화면에서 가로 스크롤되며 페이지 버튼은 줄바꿈될 수 있다.
- 기존 오늘의 파이프라인, 성과 리포트, 외부 영상 링크는 유지한다.

## 데이터 갱신과 오류 처리

- 최초 로딩에서는 목록별 로딩 표시를 사용한다.
- 자동 갱신 실패 시 기존 화면 데이터는 유지하고 해당 목록에 오류 안내를 표시한다.
- 네트워크 요청은 목록별로 독립 처리해 한 API 장애가 다른 카드 표시를 막지 않게 한다.
- 컴포넌트 언마운트 또는 페이지 변경 시 오래된 응답이 새 페이지 상태를 덮지 않도록 요청 취소 또는 응답 순서 보호를 적용한다.

## 테스트와 검증

### 백엔드

- 영상 첫 페이지·중간 페이지·빈 DB·범위 밖 페이지를 테스트한다.
- 실행 이력의 최신순 정렬, 손상 로그 제외, 페이지 경계를 테스트한다.
- `page=0`, `page_size=0`, `page_size=51`이 FastAPI 422를 반환하는지 테스트한다.
- 기존 health/status/report/pipeline 엔드포인트 회귀 테스트를 유지한다.

### 프론트엔드

- `npm run build`가 성공해야 한다.
- 개발 또는 배포 화면에서 두 목록이 독립적으로 이동하는지 확인한다.
- 자동·수동 갱신 후 현재 페이지가 유지되는지 확인한다.
- 모바일 폭에서 표와 페이징 컨트롤이 잘리지 않는지 확인한다.
- 운영 API의 `total_items`, `total_pages`, 목록 길이가 일치하는지 확인한다.

## 커밋과 푸시

- 백엔드 현재 브랜치 `codex/free-story-shorts-implementation`에 모니터링 API를 구현·검증한 뒤 백엔드 `main`에 병합하고 `origin/main`으로 푸시한다. 기존 스토리/CTA/운영 변경도 함께 기본 브랜치에 반영된다.
- 프론트엔드 `main`은 작업 전 깨끗한 상태를 확인하고 모니터링 UI 변경만 커밋·푸시한다.
- 각 저장소는 커밋 전 diff를 검토하고 관련 파일만 명시적으로 스테이징한다.
- 백엔드와 프론트엔드 모두 원격 `main` 푸시까지 수행해 GitHub 기본 화면과 연결된 배포에 실제 반영되게 한다.
- `.env`, `credentials/`, `data/`, 로컬 분석·임시 폴더는 "전체 파일" 커밋 대상에서 제외한다.
