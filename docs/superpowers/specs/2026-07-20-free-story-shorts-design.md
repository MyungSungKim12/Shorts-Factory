# 무료 스토리형 Shorts 전환 설계

## 목표

현재의 무료 자동화 자산(Gemini/Groq, Pexels/Pixabay, ffmpeg, YouTube API)을 유지하면서 채널을 랭킹 나열형에서 **실재 장소·자연현상의 비밀을 설명하는 단일 소재 스토리형 Shorts**로 전환한다.

- 첫 3초에 강한 시각·언어적 모순을 제시한다.
- 60~75초 동안 2~4초 간격으로 시각적 변화를 제공한다.
- 하루 1편으로 시작하고 샘플 및 실제 성과가 확인되면 하루 2편까지 확장한다.
- 샘플 승인 전에는 업로드·서버 스케줄·기존 데이터에 영향을 주지 않는다.
- 설정만으로 기존 랭킹 포맷과 gTTS 제작 방식으로 즉시 원복할 수 있다.

## 비목표

- 유료 생성형 이미지·영상 API 도입
- 참고 Shorts의 화면, 음성, 대본 또는 애니메이션 복제
- 샘플 검수 전 자동 공개 업로드
- 기존 업로드 영상 및 SQLite 성과 데이터 변경

## 콘텐츠 방향

채널의 핵심 문장은 다음과 같다.

> 세상에 실제로 존재하는 이상한 장소와 구조물이 왜 그렇게 만들어졌는지 1분 안에 설명한다.

초기 30편의 소재 비율은 다음을 목표로 한다.

- 실재 장소·자연현상 70%
- 역사적 구조물과 생활의 지혜 20%
- 동물의 생존 원리 10%

주력 소재는 무료 스톡 영상으로 실제 모습을 보여줄 수 있어야 한다. 최신 연예인·기업·스포츠 순위, 영화·방송 클립, 사건 CCTV, 정밀 내부 기계처럼 무료 소스로 정확히 표현하기 어려운 주제는 제외한다.

## 스토리 구조와 잔존 장치

한 편은 60~75초, 7~10개 스토리 비트와 비트당 2~3개 시각 샷으로 구성한다.

1. 0~3초: 결과 또는 모순을 먼저 제시한다. 인사·채널 로고·주제 소개는 넣지 않는다.
2. 3~12초: 장소·대상과 검증된 핵심 수치 하나를 공개한다.
3. 12~28초: 왜 문제가 생겼는지 원인을 설명한다.
4. 28~48초: 사람이 적응했거나 자연이 작동하는 방식을 보여준다.
5. 48~65초: 가장 의외인 사실 또는 현재의 결과를 공개한다.
6. 마지막 2~4초: 첫 문장을 회수하는 결론을 두고, CTA는 필요한 경우 한 문장만 사용한다.

12~15초, 25~30초, 45~50초 부근에는 새 질문·수치·장면 전환으로 주의를 다시 잡는 패턴 인터럽트를 둔다. 결론을 제목이나 첫 장면에서 모두 노출하지 않지만, 낚시가 되지 않도록 첫 10초 안에 작은 답 하나를 제공한다.

## 입출력 계약

### topic.json

스토리 포맷은 기존 순위 계약을 삭제하지 않고 별도 계약으로 추가한다.

```json
{
  "format": "story",
  "topic": "사막 한가운데 호수가 마르지 않는 이유",
  "category": "place_nature",
  "hook_angle": "비가 거의 오지 않는데 호수는 사라지지 않는다",
  "target_keyword": "desert lake",
  "core_question": "물이 어디에서 공급되는가",
  "facts": [
    {
      "claim": "검증된 사실",
      "value": "수치 또는 기록",
      "source": "기관·문헌명",
      "source_url": "https://example.com/source"
    }
  ],
  "visual_plan": [
    {"beat": "hook", "keywords": ["desert lake aerial", "dry desert water"]}
  ],
  "verification_method": "grounded_search",
  "verified_at": "2026-07-20T00:00:00+09:00"
}
```

### script.json

스토리 비트는 한 번에 자연스럽게 읽을 내레이션과 여러 시각 검색어를 함께 가진다.

```json
{
  "format": "story",
  "title": "제목",
  "description": "설명과 출처 요약",
  "tags": ["태그"],
  "hook": "첫 3초 문장",
  "scenes": [
    {
      "n": 1,
      "role": "hook",
      "narration": "짧은 내레이션",
      "visuals": ["desert lake aerial", "cracked desert ground"],
      "duration_sec": 7,
      "emphasis": ["사막", "마르지 않습니다"]
    }
  ],
  "cta": "",
  "total_duration_sec": 68
}
```

기존 `TopicContract`, `ScriptContract`, 랭킹 프롬프트 및 순위 오버레이는 그대로 보존한다. `CONTENT_FORMAT=story|ranking` 설정으로 계약과 실행 경로를 선택한다.

## 무료 미디어 제작

### 소스 선택

- Pexels 비디오 → Pixabay 비디오 → Pexels 이미지 순의 기존 폴백은 유지한다.
- 검색 결과 한 건만 무조건 선택하지 않고 여러 후보를 받아 세로 비율, 해상도, 중복 여부를 기준으로 선택한다.
- 같은 소스 ID를 한 영상에서 반복 사용하지 않는다.
- 비트마다 2~3개의 검색어를 사용하고, 한 샷은 약 2~4초 유지한다.
- 정확한 장면이 없으면 무관한 일반 영상 대신 동일 장소·환경의 와이드/디테일 샷으로 폴백한다.
- `produce_log.json`에 제공자, 소스 ID, 원본 URL, 검색어, 폴백 여부를 기록한다.

### 화면 구성

- 첫 화면은 전체 화면 영상과 2줄 이하의 훅만 사용한다.
- 기존 상단 320px 순위 띠와 누적 순위 목록은 story 포맷에서 제거한다.
- 자막은 화면 중앙 아래 안전 영역에 1~2줄로 표시한다.
- 핵심 단어·숫자는 흰색/강조색으로 분리하고 긴 문장 전체를 한 번에 보여주지 않는다.
- 정지 이미지는 느린 줌·패닝·크롭 변화로 움직임을 만든다.
- BGM은 기존 라이선스 확인된 로컬 파일만 사용하고 나레이션을 방해하지 않도록 자동 덕킹한다.

## 무료 AI 보이스

TTS를 공급자 어댑터로 분리한다.

1. 우선: Google Cloud Text-to-Speech `ko-KR-Neural2-C` 또는 사용자 선택 음성
2. 폴백: 현재 gTTS

Google Cloud Neural2는 월 100만 자 무료 구간이 있어 하루 1~2편 예상 사용량을 충분히 수용한다. 단, Google Cloud 결제 계정 연결 및 Text-to-Speech API 활성화가 필요하다. 로컬 인증은 서비스 계정 키 파일 대신 Application Default Credentials(ADC)를 사용한다. ADC 인증이 없거나 호출이 실패하면 업로드를 중단하지 않고 gTTS로 폴백하되, 샘플 로그에 실제 사용 공급자를 명확히 기록한다.

자연스러움을 위해 다음 원칙을 적용한다.

- 내레이션을 문장 하나씩 잘게 합성하지 않고 스토리 비트 단위로 합성한다.
- 쉼표·마침표·질문형을 대본에서 의도적으로 사용한다.
- 기본 말하기 속도는 TTS 자체에서 약 1.03~1.08로 조절한다.
- ffmpeg `atempo=1.2` 강제 배속은 story 포맷에서 사용하지 않는다.
- 숫자·단위 발음 치환은 기존 로직을 재사용한다.
- `TTS_PROVIDER`, `TTS_VOICE`, `TTS_SPEAKING_RATE`를 설정 가능하게 한다.

서버가 1GB 메모리인 점을 고려해 MeloTTS 같은 로컬 신경망 모델은 첫 버전에서 제외한다. 모델 다운로드·추론 메모리·배포 복잡도가 무료 운영의 안정성을 해칠 가능성이 크다.

## 샘플 안전장치

샘플은 업로더를 호출하지 않는 전용 생성 모드로 만든다.

- `scripts/generate_sample.py`가 리서처 → 작가 → 프로듀서까지만 실행한다.
- 결과 위치는 `data/samples/<sample-id>/output.mp4`로 고정한다.
- 함께 저장할 파일: `topic.json`, `script.json`, `produce_log.json`, `output.mp4`.
- 샘플 완료 후 해상도, 길이, 오디오 스트림, 검은 화면, 미디어 중복, 폴백 비율을 자동 검사한다.
- 사용자 승인 전 `run_uploader`와 서버 cron은 변경하지 않는다.

## 원복 전략

원복은 데이터 삭제나 Git 강제 초기화 없이 가능해야 한다.

1. 코드는 기능 브랜치에서 작업한다.
2. 기존 랭킹 경로와 계약을 삭제하거나 덮어쓰지 않는다.
3. `CONTENT_FORMAT=ranking`, `TTS_PROVIDER=gtts`로 설정하면 현재 제작 방식으로 복귀한다.
4. story 전용 스키마와 오버레이는 별도 모듈로 분리한다.
5. 샘플 파일은 `data/samples/`에 격리하고 기존 `data/work/`, DB, 업로드 기록을 수정하지 않는다.
6. 서버 배포와 cron 변경은 샘플 승인 후 별도 단계에서 수행한다.

## 하루 1편에서 2편으로 확장

첫 10~14편은 하루 1편만 운영한다. 다음 조건을 확인한 뒤 하루 2편을 검토한다.

- 제작 실패율과 검은 배경 폴백이 허용 수준 이내
- 사람이 검수할 수 있는 시간 내에 두 편을 확인 가능
- 기존 채널 내부 비교에서 첫 3초 유지 및 평균 시청 지속이 개선
- 두 번째 업로드가 첫 번째 영상의 초기 배분을 방해하지 않는 시간 간격 확보

공개 조회수·좋아요·댓글만 수집하는 현재 분석가로는 잔존율을 측정할 수 없다. 후속 단계에서 YouTube Analytics 권한을 추가해 `viewed vs swiped`, 평균 시청 지속시간, 평균 시청률을 수집해야 한다. 샘플 단계에서는 구현 범위에 포함하지 않는다.

## 테스트와 완료 기준

- 기존 랭킹 계약 및 테스트가 모두 통과한다.
- story topic/script 정상·누락·자리표시자·검증 방식 테스트를 추가한다.
- TTS 공급자 선택과 gTTS 폴백을 네트워크 없이 단위 테스트한다.
- 동일 미디어 중복 방지와 폴백 기록을 테스트한다.
- 샘플 실행이 업로더를 호출하지 않는지 테스트한다.
- 결과 MP4가 1080x1920, 60~75초, H.264/AAC, 오디오 포함 조건을 만족한다.
- `produce_log.json`에 실제 TTS 공급자와 모든 미디어 출처가 기록된다.
- 사용자가 전달받은 로컬 경로에서 샘플을 재생해 직접 검수할 수 있다.

## 사용자에게 필요한 작업

코드 전환과 gTTS 폴백 샘플 제작 전 필수 작업은 없다. Google Cloud Neural2의 자연스러운 보이스로 최종 샘플을 만들기 위해서는 Text-to-Speech API와 결제를 활성화하고, 로컬에서는 `gcloud auth application-default login`과 quota project 설정으로 ADC를 준비한다. 서비스 계정 JSON 키는 만들거나 `.env`에 넣지 않는다. Oracle Cloud 서버 배포 단계에서는 장기 키 대신 Workload Identity Federation을 우선 검토하며, 서버 인증 방식은 샘플 승인 후 별도로 확정한다.

샘플 승인 후에만 사용자가 다음을 결정한다.

- 최종 채널명과 브랜딩
- 하루 1회 또는 2회 업로드 시간
- 서버 배포 및 cron 변경 승인
- YouTube Analytics 추가 권한 승인
