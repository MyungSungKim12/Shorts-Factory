# 파이프라인 안정화 및 다음 회차 사전제작 설계

## 목표

무료 미디어 원본 때문에 서버 메모리와 제작 시간이 폭증하는 문제를 막고, 서로 다른 예약 회차의 동시 실행을 차단한다. 완성 영상은 업로드 직전에 구조·오디오·인트로·CTA를 검사하며, 작업 중 생성한 검증 영상은 버리지 않고 다음 예약 회차에 정시 업로드할 수 있게 한다.

## 범위

- 무료 미디어 후보의 해상도 선택 개선, 크기 제한, 스트리밍 다운로드
- 모든 예약 회차가 공유하는 전역 파이프라인 잠금과 단계별 시간 제한
- 업로드 직전 품질 게이트
- 전용 임시폴더의 정상·실패·비정상 종료 정리
- 검증 완료 영상을 다음 미래 회차로 원자적으로 승격하는 사전제작 흐름
- 기존 11:00·17:00·21:00 KST 일정과 일 3건 운영 유지

## 1. 제한된 무료 미디어 다운로드

미디어 후보는 가장 큰 파일이 아니라 1080×1920 출력에 충분하면서 목표 해상도에 가장 가까운 변형을 우선한다. 세로 영상은 720×1280 이상을 우선하고 1080×1920 이하 변형을 가장 높게 평가한다. 그 범위가 없으면 충분한 후보 중 가장 작은 변형을 사용한다.

기본 제한은 영상 80MiB, 이미지 15MiB다. `Content-Length`가 제한을 넘으면 본문을 받기 전에 후보를 거절한다. 크기를 알 수 없는 응답은 1MiB 이하 청크로 스트리밍 저장하고 누적 크기가 제한을 넘는 즉시 부분 파일을 삭제한다. 연결 10초, 읽기 30초 제한을 사용하며 실패하면 다음 후보로 이동한다. 제한은 `MEDIA_MAX_VIDEO_BYTES`, `MEDIA_MAX_IMAGE_BYTES`, `MEDIA_CONNECT_TIMEOUT_SEC`, `MEDIA_READ_TIMEOUT_SEC` 환경변수로만 조정한다.

다운로드 메타데이터에는 실제 바이트 수, 선택 해상도, 제한으로 건너뛴 후보 수를 기록한다. 비밀값과 다운로드 URL 전체는 로그에 남기지 않는다.

## 2. 전역 잠금과 실행 시간 제한

기존 `YYYYMMDD-N.lock`은 같은 회차의 중복 실행을 계속 막는다. 추가로 `data/recovery/pipeline.lock`을 모든 회차가 공유한다. 잠금 파일에는 PID, run_id, 시작 시각을 JSON으로 기록한다.

잠금 소유 프로세스가 살아 있으면 다른 회차는 FFmpeg나 업로더를 시작하지 않는다. 최대 대기 시간은 기본 90분이며 30초 간격으로 잠금을 다시 확인한다. 소유 프로세스가 없으면 잠금을 회수한다. 대기 제한을 넘으면 해당 회차를 `exhausted`로 기록하고 업로드하지 않는다.

외부 명령은 다음 기본 시간 제한을 갖는다.

- 개별 샷 FFmpeg: 180초
- 최종 합성 FFmpeg: 900초
- ffprobe 및 품질검사 FFmpeg: 180초

시간 초과는 producer 실패로 기록되어 기존 안전 재시도 규칙을 따른다. 업로더 진입 후에는 자동 재실행하지 않는다.

## 3. 업로드 직전 품질 게이트

`run_uploader`는 YouTube API와 DB를 건드리기 전에 `script.json`, `produce_log.json`, `output.mp4`를 함께 검사한다.

필수 통과 조건은 다음과 같다.

- 1080×1920, H.264 영상, AAC 오디오, 60~75초
- 영상과 오디오 스트림 길이 차이 0.5초 이하
- 검은 화면 누적 비율 10% 이하
- 영상 중간의 1.2초 이상 무음 구간 없음. 시작과 끝 0.25초는 허용
- `produce_log.script_sha256`과 현재 `script.json` 해시 일치
- `produce_log.intro.text`가 제목의 끝 문장부호만 정리한 전체 제목과 일치
- CTA 계획상 본문 내장과 별도 엔딩이 동시에 활성화되지 않음
- 별도 CTA를 붙였을 때 `produce_log.cta.text`가 script CTA와 일치하고 `구독`, `좋아요`를 모두 포함

검사 결과는 `quality_gate` 객체로 `produce_log.json`과 실행 로그에 남긴다. 한 항목이라도 실패하면 업로드를 중단하며 video insert나 YouTube API 호출은 발생하지 않는다.

## 4. 임시파일 정리

프로듀서는 시스템 임시경로 아래 `shorts-factory-` 접두사 폴더만 사용한다. 정상 반환과 예외에서는 `TemporaryDirectory`가 즉시 삭제한다. 스케줄 시작 시 전용 정리 함수가 수정된 지 6시간이 지난 동일 접두사 디렉터리만 삭제한다.

정리 전에 각 경로를 resolve하고 시스템 임시경로의 직접 자식인지, 이름이 정확한 접두사로 시작하는지 검사한다. 다른 `/tmp` 파일, 프로젝트 데이터, 현재 살아 있는 프로세스가 사용하는 폴더는 삭제하지 않는다. 삭제한 폴더 수와 총 바이트를 실행 로그에 기록한다.

## 5. 검증 영상의 다음 회차 사전제작

테스트 영상은 YouTube에 미리 비공개 업로드하지 않는다. 새 `prepare_next_slot` 흐름이 현재 KST 시각 이후 가장 가까운 11:00, 17:00, 21:00 회차를 선택하고 `data/staging/YYYYMMDD-N/`에서 리서치·대본·제작까지만 수행한다.

품질 게이트가 통과하면 다음 파일을 `data/work/YYYYMMDD-N/`으로 원자적으로 승격한다.

- `topic.json`
- `script.json`
- `output.mp4`
- `produce_log.json`
- `prepared.json`: 목표 회차, 준비 시각, 품질검사 결과, 파일 해시

대상 work 폴더에 기존 업로드 기록이나 다른 완성 파일이 있으면 승격하지 않는다. cron이 정시에 실행되면 기존 검증·해시 재사용 로직으로 리서치·대본·제작을 건너뛰고 품질 게이트를 다시 실행한 뒤 공개 업로드한다. 따라서 테스트 영상과 정규 회차 영상이 중복 생성되지 않고, 업로드 쿼터도 정시까지 사용하지 않는다.

사전제작이 다음 예약 시각까지 끝나지 않으면 그 회차에 승격하지 않는다. 실행 중인 staging 결과를 cron이 가져가지 않으며, 다음 미래 회차로 다시 지정해야 한다.

## 오류 처리와 롤백

- 크기 제한 후보는 다음 후보로 넘어가며 전체 소재를 실패시키지 않는다.
- 품질검사 실패 영상은 업로드하지 않고 work 또는 staging에 보존해 원인을 확인한다.
- 잠금 상태가 불명확하면 동시 실행보다 실행 중단을 우선한다.
- 서버 변경 전 코드, cron, DB, recovery, 대상 work/staging을 백업한다.
- `.env`와 `credentials/`는 읽기 출력·수정·커밋하지 않는다.
- 롤백 시 코드와 cron만 백업본으로 복원하며 업로드된 영상과 DB는 자동 삭제하지 않는다.

## 검증

- 적정 해상도 변형 선택과 80MiB 초과 후보 제외 테스트
- Content-Length 사전 거절과 청크 누적 초과 부분파일 삭제 테스트
- 다른 회차의 살아 있는 전역 잠금 대기 및 죽은 PID 잠금 회수 테스트
- FFmpeg 시간 초과가 producer 오류로 기록되는 테스트
- 규격, 스트림 길이 차, 중간 무음, 인트로, CTA, 해시 실패별 업로드 차단 테스트
- 전용 6시간 초과 임시폴더만 삭제하는 테스트
- 다음 KST 회차 계산, staging 실패 보존, 품질 통과 후 원자 승격 테스트
- 전체 pytest, 서버 pytest, cron·서비스·모니터링 헬스체크
- 사전제작 영상 품질 통과 후 다음 예약 회차에서 정확히 한 번 업로드되는 실제 검증
