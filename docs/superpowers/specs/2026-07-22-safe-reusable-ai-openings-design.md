# 안전한 재사용형 AI 오프닝 설계

## 목표

Vertex AI Veo의 이미지 기반 영상 생성을 스토리형 Shorts 오프닝에 도입한다. 실제 장소·동물·구조물과 화면이 다르게 만들어지는 문제를 줄이기 위해 검증된 실제 이미지를 첫 프레임으로만 사용하며, 생성 결과는 영구 자산으로 보관해 크레딧 소진 뒤에도 재사용한다.

AI 생성 실패, 할당량 부족, 크레딧 소진, 기준 이미지 부재는 업로드 중단 사유가 아니다. 모든 경우 기존 무료 스톡 오프닝으로 자동 폴백한다.

## 선택 순서

오프닝 소스는 다음 순서로 결정한다.

1. 동일한 실제 대상의 검증 완료 AI 자산을 라이브러리에서 찾는다.
2. 최근 14일 안에 사용하지 않은 자산이 있으면 재사용한다.
3. 없으면 Wikimedia 등에서 대상과 정확히 일치하고 라이선스가 확인된 실제 이미지를 구한다.
4. Veo가 활성화돼 있으면 실제 이미지를 첫 프레임으로 4초짜리 9:16 무음 영상을 생성한다.
5. 생성 결과의 첫 프레임 유사성, 영상 규격과 재생 가능성을 검사한다.
6. 통과하면 2~3초 오프닝 편집본을 만들고 영구 라이브러리에 등록한다.
7. 어느 단계든 실패하면 무료 스톡으로 진행한다.

텍스트만으로 특정 실제 장소나 랜드마크를 새로 생성하지 않는다.

## 생성 안전 규칙

- 입력은 출처·라이선스·대상 일치 여부가 확인된 실제 이미지여야 한다.
- 프롬프트는 느린 줌, 팬, 구름·물결·먼지처럼 작은 움직임만 허용한다.
- 지형, 건물, 동물의 형태를 바꾸거나 새 물체·사람·문자·로고를 추가하지 않는다.
- Veo 자체 오디오는 생성하지 않으며 기존 제목 나레이션을 사용한다.
- 모델 기본값은 `veo-3.1-fast-generate-001`, 생성 길이는 4초, 비율은 9:16이다.
- 생성 영상 첫 프레임을 추출해 기준 이미지와 지각 해시를 비교한다. 허용 범위를 넘으면 사용 금지 상태로 보관하고 스톡으로 폴백한다.
- 품질검사는 AI 영상을 실제 출처로 위장하지 않으며, 기준 이미지의 출처와 AI 생성 기록을 함께 요구한다.

## 영구 AI 자산 라이브러리

라이브러리 루트는 `data/media/ai_openings/`이다. 회차 작업물인 `data/work/`와 분리한다.

자산별 디렉터리는 다음 파일을 가진다.

- `reference.*`: 검증된 실제 기준 이미지
- `master.mp4`: Veo가 반환한 원본 4초 영상
- `opening.mp4`: 실제 Shorts에 사용하는 2~3초 편집본
- `metadata.json`: 대상, 키워드, 출처, 라이선스, 프롬프트, 모델, 생성 시각, 검증 결과

`videos.sqlite`에는 `ai_opening_assets` 테이블을 추가해 다음 정보를 색인한다.

- 안정적인 자산 ID와 실제 대상 키 `subject_key`
- 재사용 범위 `exact_subject` 또는 `concept`
- 파일 경로와 기준 이미지 출처
- 모델, 프롬프트, 생성·검증 상태
- 생성 시각, 마지막 사용 시각, 사용 횟수
- 최근 사용 이력과 실패 사유

실제 장소·동물·구조물 자산은 `exact_subject`로 등록하고 같은 대상에만 재사용한다. 폭풍, 심해 분위기, 먼지, 추상적 역사 재연처럼 특정 실물을 주장하지 않는 자산만 `concept`으로 등록해 관련 키워드가 일치하는 소재에 재사용할 수 있다. 초기 자동 생성은 안전을 위해 `exact_subject`를 기본값으로 한다.

## 영구 보존 정책

- 기존 `WORK_RETENTION_DAYS=7` 정리는 `data/work/`만 대상으로 유지한다.
- `data/media/ai_openings/`는 날짜, 사용 횟수, 검증 실패 여부와 관계없이 자동 삭제하지 않는다.
- 검증에 실패한 생성물도 `rejected` 상태로 보관하되 영상 제작 후보에서는 제외한다.
- 서버 디스크가 부족해져도 자동 삭제하지 않고 경고만 기록한다.
- 자산 삭제는 향후 명시적인 관리 명령 또는 사용자의 직접 요청으로만 허용한다.
- 작업 회차를 재생성하거나 `data/work/<run_id>`를 삭제해도 AI 자산은 영향을 받지 않는다.

현재 서버는 45GB 중 약 35GB가 남아 있다. 영구 보존을 우선하되 운영 문서에 라이브러리 용량 확인 명령을 추가한다.

## Vertex AI 연결

공식 Google Gen AI SDK의 Vertex AI 모드를 사용한다. 필요한 운영 설정은 실제 값 없이 `.env.example`에 설명한다.

- `GOOGLE_CLOUD_PROJECT`
- `GOOGLE_CLOUD_LOCATION=global`
- `GOOGLE_GENAI_USE_VERTEXAI=true`
- `VEO_OPENING_ENABLED=true`
- `VEO_MODEL=veo-3.1-fast-generate-001`
- `VEO_OPENING_MAX_SEC=3.0`
- 생성 완료 대기 제한과 폴링 간격

인증은 서버의 기존 Google ADC를 사용하고 비밀 키를 Git에 저장하지 않는다. SDK나 API 호출이 불가능하면 예외를 외부로 전파하지 않고 구조화된 실패 기록을 남긴 뒤 스톡으로 진행한다.

## 실행 기록과 비용 관찰

`produce_log.json`의 오프닝 기록에는 다음을 남긴다.

- `opening_strategy`: `ai_library`, `vertex_veo_image`, `stock_after_veo_failure`, `exact_stock`
- 자산 ID와 재사용 여부
- 실제 대상 키와 기준 이미지 출처
- 모델, 생성 길이, 사용 길이
- 생성 성공·실패·검증 거부 사유
- 해당 회차에서 신규 Veo 호출을 했는지 여부

비용은 청구 시스템의 확정 금액으로 간주하지 않고, 신규 생성 초 수를 기반으로 한 참고 추정치만 로그에 남긴다.

## 오류 처리

- 라이브러리 DB가 손상되면 파일을 삭제하지 않고 스톡으로 폴백한다.
- 생성 장기 작업이 시간 초과되면 실패로 기록하고 다음 업로드 단계를 계속한다.
- 결과 다운로드 또는 ffmpeg 검증 실패 시 원본과 오류 메타데이터를 `rejected`로 보존한다.
- 동일 대상에 대한 동시 생성은 파일 잠금으로 막아 중복 과금을 방지한다.
- 준비 영상 재시도에서는 이미 생성된 자산을 먼저 찾아 같은 Veo 요청을 반복하지 않는다.

## 검증

- 실제 기준 이미지가 없으면 Veo를 호출하지 않는 테스트
- 동일 대상의 자산이 있으면 신규 호출 없이 재사용하는 테스트
- 최근 14일 사용 자산을 피하고 가능한 다른 자산을 고르는 테스트
- Veo 실패·시간 초과·검증 실패가 스톡 폴백으로 끝나는 테스트
- 첫 프레임 유사성 기준을 통과·거부하는 테스트
- 같은 대상의 동시 생성이 한 번만 호출되는 테스트
- `cleanup_old_work()`가 7일 지난 `data/work/`는 삭제하지만 `data/media/ai_openings/`는 절대 삭제하지 않는 회귀 테스트
- 기존 제목 음성, CTA, 60~75초 길이, 품질 게이트 테스트

샘플 영상 한 건으로 실제 이미지 기반 움직임과 생성 로그를 확인한 뒤 운영 서버에 활성화한다. 서버 배포 전 `.env`, 인증, 데이터, cron을 백업하고 기존 하루 4회 스케줄은 변경하지 않는다.
