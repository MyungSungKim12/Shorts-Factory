# Shorts 파이프라인 자동 복구 및 인트로 완결성 설계

## 목표

무료 API의 불완전 응답이나 일시 장애가 발생해도 예약 회차가 가능한 범위에서 자동 복구되게 한다. 동시에 인트로가 글자 수 제한 때문에 문법적으로 잘리는 문제를 제거하고, 운영자가 모니터링 화면에서 실패와 복구 상태를 즉시 확인할 수 있게 한다.

## 범위

- 백엔드: 완결된 인트로, 대본 템플릿 폴백, 예약 회차 지연 재시도, 중복 실행 방지, 복구 상태 API
- 프론트엔드: 실행 이력에 실패 원인·재시도 횟수·다음 재시도 시각·복구 결과 표시
- 운영: 20260721-1 회차의 삭제된 YouTube 영상을 백업 후 재합성·재업로드
- 제외: 유료 API 도입, 하루 3회 예약 시간 변경, YouTube 일 6건 상한 변경

## 인트로 완결성

`_spoken_intro`는 제목을 22자에서 자르지 않는다. 제목 끝의 물음표·느낌표만 정리하고 전체 제목을 읽는다. 영상 길이는 실제 정리된 Neural2 음성 길이로 다시 계산하며, 60~75초 검증을 그대로 적용한다. 완결된 제목 때문에 75초를 넘으면 음성을 자르지 않고 제작을 중단한다.

## 대본 생성 복구

1. Groq 우선, Gemini 대체 호출을 유지한다.
2. 반환 JSON 파싱 또는 스키마 검증 실패 시 압축 JSON 지시로 한 번 재생성한다.
3. 두 모델 시도가 모두 실패하거나 두 번째 응답도 불완전하면 검증된 `topic.json`만 사용해 결정적 스토리 대본을 만든다.
4. 템플릿 대본은 `facts`와 `visual_plan` 이외의 수치·인과관계를 만들지 않는다.
5. 템플릿은 hook/context/problem/mechanism/payoff/close 역할을 포함한 8개 장면, 본문 목표 53~58초, 별도 CTA를 생성한다.
6. 사용한 경로를 `writer_mode` 값 `llm`, `llm_retry`, `verified_template` 중 하나로 실행 로그에 기록한다.

## 예약 회차 자동 재시도

기존 cron은 `scripts/run_daily.py` 대신 복구 래퍼를 호출한다. 래퍼는 회차별 잠금 파일을 원자적으로 획득해 같은 슬롯의 중복 실행을 막는다.

- 첫 실행 실패 시 실행 로그와 SQLite를 확인한다.
- 업로더 성공 또는 업로드 여부가 불명확하면 자동 재시도하지 않는다.
- researcher/writer/producer 단계의 명확한 실패이고 업로드 기록이 없을 때만 15분 뒤 동일 슬롯을 한 번 재실행한다.
- 두 번째 실패 후에는 중단하고 상태를 `exhausted`로 기록한다.
- 다음 예약 실행 시작 시 당일 이전 슬롯에 `scheduled` 또는 `running` 상태가 남아 있으면 실제 프로세스·로그를 대조해 `recovered` 또는 `exhausted`로 정규화한다.
- 하루 업로드 상한 3건과 코드 상한 6건은 유지한다.

복구 상태는 `data/recovery/YYYYMMDD-N.json`에 저장하며 `run_id`, `attempts`, `status`, `last_error`, `next_retry_at`, `updated_at`을 포함한다.

## 모니터링

백엔드 실행 이력 응답에 선택적 `recovery` 객체를 합친다. 기존 클라이언트 호환성을 유지한다. 프론트엔드는 각 실행 이력에 다음을 표시한다.

- 실패 단계와 축약하지 않은 핵심 오류
- `재시도 예정`, `재시도 중`, `자동 복구 완료`, `자동 복구 실패` 배지
- 시도 횟수와 다음 재시도 시각
- 복구 후 업로드된 영상 링크

페이지네이션과 기존 목록 데이터는 그대로 유지하며, 복구 상태 조회 실패가 영상 목록을 지우지 않게 한다.

## 20260721-1 재업로드 절차

1. 서버의 `data/work/20260721-1`, 해당 실행 로그, SQLite를 타임스탬프 백업한다.
2. 사용자가 기존 YouTube 영상을 삭제했다는 확인을 근거로 해당 회차 SQLite 업로드 행만 제거한다.
3. `output.mp4`와 `produce_log.json`만 제거해 소재와 검증된 대본은 보존한다.
4. 새 인트로 코드로 동일 회차를 재합성한다.
5. 1080x1920, H.264/AAC, 60~75초, 오디오 존재, 대체 화면 수를 검증한다.
6. 동일 회차로 한 번만 공개 업로드하고 새 video ID를 실행 로그와 SQLite에서 확인한다.

## 안전 및 롤백

- `.env`, `credentials/`, API 키는 수정하거나 출력하거나 커밋하지 않는다.
- 모든 서버 변경 전 앱 코드, cron, DB, 대상 회차 데이터를 백업한다.
- 업로드 성공이 불명확한 경우 자동 재업로드하지 않는다.
- 배포 실패 시 코드·cron·DB를 같은 백업 시점으로 복원한다.

## 검증

- 인트로가 전체 제목을 반환하는 회귀 테스트
- 불완전 JSON 두 번 후 검증 템플릿으로 전환되는 테스트
- facts/visual_plan 밖의 내용을 템플릿이 사용하지 않는 테스트
- 재시도 가능/불가능 단계와 15분 예약 계산 테스트
- 슬롯 잠금 및 최대 1회 재시도 테스트
- history API recovery 병합과 페이지네이션 회귀 테스트
- 프론트엔드 상태 배지 렌더링 및 빌드 테스트
- 서버 전체 테스트, cron 확인, 서비스 헬스체크, 실제 재합성·업로드 검증
